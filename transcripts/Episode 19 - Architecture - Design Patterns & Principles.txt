歡迎來到設計模式與原則篇。這集我們挑戰軟體工程的基石——設計原則與常見模式。這些概念將幫助你寫出更易於維護與擴充的程式碼。

首先，我們從最經典的 SOLID 原則與其他設計心法開始，接下來這是良好設計的基礎。

單一職責原則
Single Responsibility Principle
A class should have only one reason to change.
一個類別應該只有一個改變的理由。

開放封閉原則
Open Closed Principle
Software should be open for extension but closed for modification.
軟體應該對擴展開放，但對修改封閉。

里氏替換原則
Liskov Substitution Principle
Subtypes must be substitutable for their base types without breaking code.
子型別必須能替換其基底型別而不破壞程式碼。

介面隔離原則
Interface Segregation Principle
Clients should not depend on interfaces they don't actually use.
客戶端不應依賴他們實際上不使用的介面。

依賴反轉原則
Dependency Inversion Principle
High-level modules should depend on abstractions, not concretions.
高層模組應依賴抽象，而非實作細節。

不要重複自己
DRY
DRY 是 Don't Repeat Yourself 的縮寫
Refactor the code to adhere to the DRY principle.
重構程式碼以遵守 DRY 原則。

保持簡單
KISS
KISS 是 Keep It Simple, Stupid 的縮寫
Apply the KISS principle, don't overengineer the solution.
應用 KISS 原則，不要過度設計解決方案。

接下來，我們來看看其他重要的原則。

關注點分離
Separation of Concerns
Separation of concerns makes the code easier to maintain and test.
關注點分離讓程式碼更易於維護與測試。

高內聚
High Cohesion
Aim for high cohesion where related functions are kept together.
目標是高內聚，將相關的功能保持在一起。

低耦合
Low Coupling
Low coupling reduces the impact of changes on other modules.
低耦合降低了變更對其他模組造成的影響。

程式碼品質的維護，與架構設計一樣重要，接下來我們需要了解這些關於壞味道與重構的詞彙。

重構
Refactoring
Refactoring improves code structure without changing its behavior.
重構在不改變行為的前提下改善程式碼結構。

技術債
Technical Debt
We need to pay down technical debt to maintain development speed.
我們需要償還技術債以維持開發速度。

遺留程式碼
Legacy Code
Working with legacy code without tests can be challenging.
處理沒有測試的遺留程式碼極具挑戰性。

程式碼異味
Code Smell
Long methods and large classes are common code smells.
過長的方法和過大的類別是常見的程式碼異味。

設計模式提供了解決常見問題的模板，接下來我們從建立型與行為型模式看起。

單例模式
Singleton
Use the singleton pattern for the global database connection pool.
使用單例模式來處理全域資料庫連線池。

工廠模式
Factory
The factory pattern creates objects without specifying the exact class.
工廠模式在不指定確切類別的情況下建立物件。

建造者模式
Builder
The builder pattern helps construct complex objects step by step.
建造者模式協助逐步建構複雜物件。

策略模式
Strategy
Use the strategy pattern to swap sorting algorithms at runtime.
使用策略模式在執行階段切換排序演算法。

觀察者模式
Observer
The observer pattern is key to implementing event-driven systems.
觀察者模式是實作事件驅動系統的關鍵。

結構型模式則關注於物件的組成與介面適配，例如接下來幾種。

裝飾者模式
Decorator
Add functionality dynamically to an object using the decorator pattern.
使用裝飾者模式動態增加物件的功能。

轉接器模式
Adapter
The adapter allows incompatible interfaces to work together seamlessly.
轉接器允許不相容的介面無縫地協同工作。

外觀模式
Facade
Provide a simple interface to a complex system via a facade.
透過外觀模式為複雜系統提供一個簡單介面。

代理模式
Proxy
Use a proxy to control access or add logging to the object.
使用代理模式來控制存取或為物件增加日誌記錄。

儲存庫模式
Repository
The repository pattern abstracts data access logic from business logic.
儲存庫模式將資料存取邏輯從商業邏輯中抽象出來。

最後，讓我們接著看決定系統的大方向的幾種常見架構。

單體式
Monolith
The startup began with a monolithic architecture for simplicity.
這家新創公司為了簡單起見，以單體式架構起步。

分層架構
Layered Architecture
Typical layered architecture includes presentation, business, and data layers.
典型的分層架構包含展示層、商業層和資料層。

事件驅動
Event-Driven
Event-driven architecture improves scalability by decoupling services.
事件驅動架構透過解耦服務提升了可擴展性。

無伺服器
Serverless
Serverless functions scale automatically based on the request volume.
無伺服器函式會根據請求量自動擴展。

網域驅動設計
DDD
DDD 是 Domain-Driven Design 的縮寫
DDD focuses on the core business logic and domain model.
DDD 專注於核心商業邏輯與領域模型。


設計模式是前人智慧的結晶。當你遇到架構難題時，回頭看看這些經典模式，往往能找到靈感。感謝收聽。
