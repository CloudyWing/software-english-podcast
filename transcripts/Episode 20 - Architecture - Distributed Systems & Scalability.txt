歡迎來到分散式系統架構篇。掌握了設計模式後，我們要將視野擴大到整個系統架構。這集涵蓋了分散式系統、擴展策略以及相關的通訊模式。

分散式系統的核心挑戰，在於資料的一致性與可用性，這個領域有許多理論與模型。

CAP 定理
CAP Theorem
CAP 是 Consistency, Availability, Partition Tolerance 的縮寫
The CAP theorem states you can only pick two: Consistency, Availability, Partition tolerance.
CAP 定理指出你只能三選二：一致性、可用性、分區容錯性。

最終一致性
Eventual Consistency
We accept eventual consistency for the user news feed.
我們接受使用者動態牆採用最終一致性。

分散式交易
Distributed Transaction
Managing distributed transactions across microservices is complex.
管理跨微服務的分散式交易很複雜。

命令查詢職責分離
CQRS
CQRS 是 Command Query Responsibility Segregation 的縮寫
CQRS separates read and write operations into different models.
CQRS 將讀取與寫入操作分離成不同的模型。

事件溯源
Event Sourcing
Event sourcing stores the state changes as a sequence of events.
事件溯源將狀態變更儲存為一系列的事件。

系統的擴展性決定了它能服務多少使用者，主要有兩種擴展方向。

水平擴展
Scale Out
Add more servers to scale out the system to handle load.
增加更多伺服器來水平擴展系統以處理負載。

垂直擴展
Scale Up
Add more RAM or CPU to scale up the single server.
增加記憶體或 CPU 來垂直擴展單一伺服器。

快取
Caching
Caching strategies drastically improve read performance.
快取策略大幅提升讀取效能。

為了確保系統穩定，我們需要強大的韌性設計，防止單點故障導致全面崩潰。

高可用性
High Availability
Design for high availability to minimize downtime during failures.
為高可用性設計以最小化故障期間的停機時間。

容錯
Fault Tolerance
The system has built-in fault tolerance to handle unexpected crashes.
系統內建容錯機制以處理預料之外的當機。

冗餘
Redundancy
Data redundancy protects against hardware failure.
資料冗餘可保護資料免受硬體故障影響。

斷路器
Circuit Breaker
Implement a circuit breaker to prevent cascading failures in the system.
實作斷路器以防止系統中的連鎖故障。

服務間的非同步溝通，通常依賴訊息傳遞機制，以下是核心組件。

訊息佇列
Message Queue
Use a message queue like RabbitMQ to decouple services.
使用像 RabbitMQ 這樣的訊息佇列來解耦服務。

發布訂閱
Pub Sub
Pub Sub 是 Publish-Subscribe 的縮寫
The Pub Sub model allows asynchronous communication between components.
發布訂閱模型允許元件間進行非同步溝通。

代理人
Broker
The message broker handles routing messages to the correct consumers.
訊息代理人負責將訊息路由至正確的消費者。


分散式系統充滿了取捨。在追求高效能與高可用性的同時，千萬別忘了系統的複雜度也會隨之上升。感謝收聽。
